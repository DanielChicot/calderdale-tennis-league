# Phase 2 prep — open questions and carry-overs

> Brainstorming seed for Phase 2 (scraper + snapshot pipeline + live API). Written at the end of Phase 1 so the next session — possibly on a different machine — doesn't have to re-derive context from chat history.

## Open questions to resolve in Phase 2 brainstorming

These each have a non-obvious answer and a decision should be made before plan-writing:

1. **JSON snapshot storage: R2 vs git-tracked `data/` directory.** Marked open in the Phase 1 design spec. R2 is the spec's default; git-tracked is simpler and zero-cost for a personal project but bloats the repo over time. Trade-off matters for hosting costs, debuggability via diffs, and machine-B onboarding.

2. **Scraping cadence.** "Hourly during the season, daily off-season" was the original handwave. What defines "the season"? Manual config knob, auto-detect from current-season metadata, or always hourly and stop worrying about it?

3. **Archive scope.** Scrape every historical season (Summer 2021–2025 + future) on every run, or scrape current-only on schedule and run archive scrape as a one-off / on-demand?

4. **Entity resolution / name normalisation.** The upstream uses two different names for the same club ("Halifax Queens" in player rankings, "Queens Sports Club" in the clubs directory). Phase 2 needs to decide where this maps to a single canonical entity — at parse time, at scrape time, or at consumer time?

5. **Polite-scraping etiquette.** Rate-limit (1 req/sec was the spec default), retry policy, robots.txt respect, ETag / If-Modified-Since for unchanged pages, User-Agent contact address (currently personal email — fine for now, swap before public).

6. **Live-API Worker scope.** Should the Worker just serve volatile bits (today's results, current standings) and let snapshots handle everything else, or also act as a fallback when R2 is stale / unavailable? Determines complexity and edge-case handling.

## Carry-overs from Phase 1 final review

The whole-phase reviewer flagged these. None blocked Phase 1 shipping but several deserve fixing in Phase 2:

- **[FIXED in `16ce9f2`]** `Ranking.rubbersWon` / `rubbersPlayed` were `z.number().int()` but real data has half-points. Dropped `.int()`.
- **[LOW]** `Club.id` is insertion-order positional, not stable across scrapes. Snapshot pipeline needs a stable-id strategy (slug-based hash, or upstream identifier if discoverable).
- **[LOW]** Asymmetry: `parseClubsDirectory` returns domain `Club[]`, `parseLeagueTable` / `parsePlayerRankings` return intermediate row types. Phase 2 should decide explicitly what the new parsers return.
- **[LOW]** `detectPageType` currently only knows about `tabIndex=0` and `tabIndex=4` shell URLs. Phase 2 fragment URLs (`displayResults.php`, `result_card_*.php`, etc.) will need their own detection logic. Decide: extend the function, split into shell-detector + fragment-detector, or add a separate `detectFragmentType`?
- **[LOW]** `undici` is pinned at v8 in root devDeps and v6 in `packages/parser` deps. Reconsider whether parser needs undici at all — Node 24 has built-in `fetch`.
- **[LOW]** All three parsers `import * as cheerio from 'cheerio'`. Only `cheerio.load` is used. Tree-shake-friendlier `import { load } from 'cheerio'` would set a better precedent for the new parsers.

## Domain / upstream lore worth remembering

- **Upstream architecture is hybrid.** Most tabs are server-rendered inline in the `index.php` shell; only tab 1 (Fixtures & Results) loads via XHR. Full discovery in `spike/fragment-urls.md`.
- **Two club names for the same entity.** Clubs directory: "Queens Sports Club". Player rankings: "Halifax Queens". Almost certainly the same place (a club's full legal name vs the team's display name). Phase 2 entity resolution will surface more of these.
- **Modal IDs.** `modeID=3` is Mixed Division 1 (not Mens). Full map of division IDs is in `spike/fragment-urls.md`. The fixture filenames in `fixtures/` use the corrected `-mixed-` naming.
- **Tennis half-points are real.** Both team points (`pointsWon`, `pointsLost`) and player rubber counts (`rubbersWon`) can be `.5`. The parser uses `parseDecimalStrict` for these fields. Don't assume integer in any new parser.
- **CSRF is inert.** `refreshProtectionCode=0` works everywhere we've tested. Plain `fetch` with a polite UA is sufficient. See `spike/findings.md`.
- **`refreshProtectionCode` for `displayContacts.php`, `displayLocations.php`, `result_card_*.php` is unverified.** Spike confirmed the param is inert for `displayResults.php` and shell pages, but didn't fully test the detail-fragment endpoints. May need a session warm-up for these — Phase 2 scraper should probe early.

## Proposed Phase 2 scope (to confirm in brainstorming)

- New parsers for the fragment endpoints:
  - `displayResults.php` (fixtures + results for a division)
  - `result_card_*.php` (per-fixture match cards)
  - `displayContacts.php` (per-team contact details)
  - `displayLocations.php?Mode=html` (per-club location details)
- A `parser` package for the season/archive nav (discover divisions + season IDs)
- `apps/scraper` — orchestrator that walks the upstream, calls parsers, writes JSON
- Storage backend decision per Q1 above, with whatever wiring it needs
- `.github/workflows/scrape.yml` — cron-triggered scrape
- Smoke / regression test: parser snapshots refresh script with diff-alerting

## Out of scope for Phase 2 (already deferred)

- Live-night auto-refresh mode (Phase 7 or later)
- Streaks & notable runs
- Personalised dashboard (Phase 5)
- Web frontend (Phase 4)
- Public launch / domain branding / league blessing
